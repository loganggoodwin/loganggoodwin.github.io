This project evaluates the existing network infrastructure of SNHUEnergy, Inc. — a multi-site energy company with offices in Dallas and Memphis — identifies performance and security deficiencies, and proposes a dual-hub SD-WAN architecture to support expansion into Kansas City and Houston. It covers OSI-layer analysis, traffic characterization, security hardening, and a phased implementation plan.

Current state assessment

SNHUEnergy's existing infrastructure supports billing, accounting, HR, and operations systems across two sites. Several critical weaknesses were identified during the assessment.

High severity

Memphis lacks a local firewall, leaving its segment dependent on Dallas for perimeter security — a single point of failure for the entire WAN.

High severity

Both sites rely on a single WAN connection. No redundancy means any provider outage interrupts operations across all applications.

Medium severity

No QoS implementation means VoIP and video traffic competes equally with bulk data, degrading call and conference quality under load.

Medium severity

Insufficient capacity management creates bottlenecks for latency-sensitive applications like SQL queries, payroll, and billing workflows.

Proposed architecture: dual-hub SD-WAN

The recommended design establishes Dallas and Houston as redundant SD-WAN hubs, with Memphis and Kansas City as branch spokes. This eliminates both the single-hub and single-path weaknesses in the current infrastructure.

WAN and connectivity

  • Dual WAN links at each site — primary MPLS and backup internet — with SD-WAN handling automatic failover and path selection
  • End-to-end IPsec VPN encryption across all inter-site tunnels
  • SSL VPN gateways embedded in NGFWs for secure remote access
  • SD-WAN tunnels for Office 365, Azure, and AWS workloads to avoid backhauling cloud traffic through hub sites

LAN segmentation

  • VLAN separation for Finance, HR, and Operations within each site's LAN
  • Layer 3 switches with 802.1Q trunking and QoS tagging between departments
  • Role-based access controls aligned to VLAN boundaries
  • QoS policies prioritizing VoIP and video over bulk data transfers

Security controls

  • Next-generation firewalls at every site — application-layer inspection, IPS, SSL decryption, and geo-blocking
  • Web application firewall protecting internet-facing services from injection and XSS attacks
  • UTM appliances merging antivirus, content filtering, and spam protection
  • Multi-factor authentication required for all remote and cloud access

Monitoring and visibility

Tooling recommendation SolarWinds NPM provides real-time SNMP-based performance data across all sites from a single dashboard, enabling proactive alerting before degradation impacts operations. Splunk Enterprise Security aggregates logs from firewalls, servers, and endpoints for incident correlation and forensic analysis — shifting network management from reactive to predictive.

Disaster recovery design

  • Houston serves as the secondary hub — Active Directory, DNS, and database servers replicate to Houston via daily secure replication
  • If Dallas experiences an outage, Houston seamlessly takes over essential services
  • SD-WAN's diverse path selection keeps inter-office communication online even during provider outages
  • Recovery time and recovery point objectives defined in the continuity plan

Scalability

New branch offices — regional sales or additional data centers — can be onboarded using SD-WAN's zero-touch provisioning. Global security policies, QoS rules, and routing configurations are applied automatically at provisioning time, significantly reducing deployment time and configuration drift.

OSI layer analysis

The project maps each application and infrastructure component to its relevant OSI layers, connecting abstract network concepts to concrete operational decisions. Physical layer redundancy (dual WAN), data link layer trunk configuration (802.1Q), network layer routing (OSPF/BGP and SD-WAN), transport layer QoS prioritization, and application layer firewall inspection are each addressed in the architecture.
