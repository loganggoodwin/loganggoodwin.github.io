Password Hash Cracking: NTLM and MD5 Analysis

Projects Home

This write-up documents my approach to password hash cracking challenges in the Cyber Skyline CTF competition. The challenges involved identifying hash types, selecting the right cracking strategy, and recovering plaintext passwords from NTLM and MD5 hashes — skills that directly mirror real-world credential analysis during incident response and penetration testing engagements.

Challenge overview

The challenge set presented a series of password hashes without labels. The first task was identifying hash types from structure alone, then applying the appropriate cracking approach for each. No hints were given about the algorithm, password length, or complexity.

What this tests Hash cracking in CTF mirrors real incident response work — when an attacker dumps credentials from a compromised Windows host, defenders need to understand what those hashes represent, how quickly they can be cracked, and what that tells them about an organization's password policy.

Hash identification

Before cracking anything, you need to know what you are looking at. Hash length and character set narrow it down significantly.

Hash type Length Character set Common source
MD5 32 hex chars a–f, 0–9 Web apps, older Linux /etc/shadow
NTLM 32 hex chars a–f, 0–9 Windows SAM database, Active Directory
SHA-1 40 hex chars a–f, 0–9 Git commits, older certificates
bcrypt 60 chars Base64 + $2a$ prefix Modern web apps, Linux PAM

MD5 and NTLM are both 32-character hex strings, which makes them visually identical. The distinction matters because their cracking speed differs by orders of magnitude — NTLM is unsalted and extremely fast to crack, while MD5 context (salted vs. unsalted) changes the approach entirely. Tools like hashid or hash-identifier can disambiguate when structure alone is not enough.

# Quick hash identification
hashid 5f4dcc3b5aa765d61d8327deb882cf99

Analyzing '5f4dcc3b5aa765d61d8327deb882cf99'
[+] MD2
[+] MD5     <-- most likely given length and context
[+] MD4

Methodology

My approach followed a consistent escalation path: start fast and cheap, move to expensive only when needed.

  1. Check against known databases first Paste the hash into a lookup tool like CrackStation or hashes.com. Common passwords like password, 123456, and letmein appear in precomputed tables. This takes seconds and costs nothing.
  2. Run a wordlist attack with rules Feed the hash into Hashcat or John the Ripper with rockyou.txt and a ruleset. Rules handle common mutations — capitalization, number appending, leet substitutions — without brute-forcing every combination.
  3. Targeted mask attack if wordlist fails If you can infer password policy (e.g., minimum 8 chars, must contain a number), a mask attack narrows the keyspace significantly versus full brute force.
  4. Document what worked and why Recording which approach cracked which hash tells you something about the underlying password policy — useful for the challenge debrief and real-world parallels.

NTLM cracking

NTLM is Windows' legacy password hashing algorithm. It applies no salt and performs a single MD4 pass over the UTF-16LE encoded password. This makes it extremely fast to crack — modern GPUs can test billions of NTLM candidates per second.

# Hashcat — NTLM is mode 1000
hashcat -m 1000 -a 0 hashes.txt rockyou.txt

# Add rules for common mutations
hashcat -m 1000 -a 0 hashes.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rule

# Mask attack — 8 char, uppercase first, 6 lowercase, 1 digit
hashcat -m 1000 -a 3 hashes.txt ?u?l?l?l?l?l?l?d
Why NTLM matters in real investigations When attackers dump the Windows SAM database or NTDS.dit from Active Directory, they get NTLM hashes. The speed at which these crack is why password length policies and account lockout controls matter — a 6-character NTLM hash can fall in seconds. This is also why Pass-the-Hash attacks are so dangerous: you do not always need the plaintext.

MD5 cracking

MD5 is mode 0 in Hashcat. Unsalted MD5 cracks at similar speeds to NTLM. The challenge here was confirming the hash was unsalted — salted MD5 requires a different mode and the salt value itself.

# Hashcat — raw MD5 is mode 0
hashcat -m 0 -a 0 hashes.txt rockyou.txt

# John the Ripper — auto-detects format
john --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt

# Show cracked results
john --show hashes.txt

What I learned

  • NTLM's lack of salting is not an accident of history — it was a design choice that made network authentication faster at the cost of storage security. Understanding the tradeoff matters for recommending mitigations.
  • Wordlist quality beats brute force almost every time. rockyou.txt contains 14 million real passwords from actual breaches — attackers know what people actually choose.
  • Rule-based mutations (appending !, capitalizing first letter, substituting @ for a) recover passwords that survive a plain wordlist attack. Most password policies inadvertently encourage these patterns.
  • Hash identification is a skill in itself. Automated tools help but developing intuition for hash structure — length, prefix, character set — speeds up triage during time-limited competitions and real incidents.
  • The difference between cracking speed for NTLM versus bcrypt is the entire argument for why modern systems should use adaptive hashing algorithms like Argon2 or bcrypt with appropriate cost factors.

Defensive takeaways

The offensive side of this challenge maps directly to defensive recommendations I would make in an IT support or junior security role:

What this means for defenders Enforce minimum 12-character passwords. Avoid complexity rules that encourage predictable substitutions — length is a stronger control. Enable MFA on all accounts so cracked hashes do not immediately translate to access. Audit for NTLM usage and migrate to NTLMv2 or Kerberos where possible. Monitor for Pass-the-Hash indicators in Windows event logs (Event ID 4624, logon type 3 with NTLM).
